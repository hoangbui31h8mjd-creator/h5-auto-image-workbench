# 商单 H5 自动生成工具 PRD v0.1

> 文档状态：初版草案  
> 当前阶段：工作流验证 / Demo 演示  
> 目标方向：后续包装成内部 Web 工具或轻量产品化平台  
> 更新时间：2026-07-01

---

## 1. 项目背景

商单 H5 长图制作通常包含需求理解、文案整理、原型排版、视觉设计、长图切片、生图、拼接和人工修改等环节。当前流程依赖设计师经验和多轮沟通，尤其在活动页结构相似、交付周期紧的场景中，重复劳动较多。

本项目希望验证一条可自动化的生产链路：业务方输入 KV 图、活动文案和技术要求后，系统自动生成 H5 原型图、裁切方案、原型切片，并调用图像模型分段生成最终 H5 长图。

当前项目不承诺无人精修生产交付，核心目标是先验证链路是否可跑通、关键技术卡点是否有解法。

## 2. 产品定位

商单 H5 自动生成工具是一个面向运营、设计和项目协作场景的内部自动化工具。

它不是普通的随机生图工具，而是一个受控的 H5 生产工作流：

```text
输入 KV / 文案 / 约束
-> 生成结构蓝图
-> 生成 HTML wireframe 原型
-> 渲染完整原型图
-> 智能裁切为 1-N 段
-> 分段生图
-> 接龙拼接
-> 输出完整 H5 长图与中间产物
```

## 3. 目标用户

### 3.1 运营 / 业务方

诉求：

- 快速提交活动需求。
- 获得第一版可讨论的 H5 长图方案。
- 降低反复描述需求和等待设计排期的成本。

### 3.2 设计师 / 创意同学

诉求：

- 减少重复搭建活动页结构的工作。
- 快速获得原型、切片和初步视觉结果。
- 把时间更多投入到精修、风格判断和关键创意上。

### 3.3 项目负责人 / 管理者

诉求：

- 验证 AI 自动化链路的可行性。
- 评估商单物料生产效率提升空间。
- 后续形成可复用的平台化能力。

## 4. 产品目标

### 4.1 当前 Demo 阶段目标

- 用户可以上传 KV 图，填写活动文案和技术要求。
- 系统可以生成结构化蓝图和 HTML wireframe。
- 系统可以渲染完整原型图并自动规划 1-N 段切片。
- 系统可以调用 ComfyUI / Image2 完成 mock 或真实分段生图。
- 系统可以返回最终长图和关键中间产物。
- 形成可用于汇报演示的完整链路。

### 4.2 产品化阶段目标

- 将 Dify 工作流能力迁移或封装为稳定后端服务。
- 提供前端页面供业务方自助提交需求。
- 支持任务状态追踪、结果预览、失败重试和人工审核。
- 支持中间产物管理，包括蓝图、HTML、原型图、切片、最终图。
- 增加自动质检，包括尺寸、切点、模块完整性、文字、接缝等。

## 5. 不做什么

当前阶段暂不承诺：

- 不承诺最终图达到设计师精修水平。
- 不承诺所有商单类型都能稳定生成。
- 不承诺一次生成即可上线。
- 不做复杂权限、审批、团队协作和计费系统。
- 不做完整 H5 页面开发和前端可交互发布，只输出长图。

## 6. 核心流程

### 6.1 用户提交

用户在页面中提交：

- KV 图。
- 活动文案。
- 技术要求。
- 可选：目标语言、活动类型、希望风格、交付尺寸、参考案例。

### 6.2 原型生成

系统调用 LLM 和知识库，生成：

- 页面结构蓝图。
- 模块顺序。
- 文案映射。
- 候选 safe_bands。
- 自包含 HTML wireframe。

约束：

- HTML 宽度统一 750px。
- KV 只用于理解风格，不直接嵌入原型。
- 原型必须是黑白灰低保真 wireframe。
- 原型不得显示安全区、切口、接缝、40px 等生产术语。

### 6.3 渲染与裁切

ImageService 完成：

- HTML 渲染成完整长原型图。
- 根据页面高度、模块边界、低复杂度区域和单段最大高度规划切点。
- 输出 1-N 张原型切片。

约束：

- 不固定三段。
- 优先避免切断标题、按钮、表格、视频框、奖励卡、人物、商品等主体内容。
- 当前单段最大目标高度约 2250px。

### 6.4 分段生图

ComfyUI 完成：

- 读取 KV 图和 N 张原型切片。
- 进行业务意图分析。
- 为每段生成提示词。
- 调用 Image2 逐段生成。
- 第 2 段开始使用上一段底部约 40px 作为续画参考。

### 6.5 拼接与输出

系统完成：

- 对相邻段进行重叠羽化拼接。
- 输出最终 H5 长图。
- 返回完整原型图、切片、裁切计划和调试信息。

## 7. 功能需求

### 7.1 需求提交页

优先级：P0

字段：

- KV 上传：必填，支持 jpg/png。
- 活动文案：必填，多行文本。
- 技术要求：必填或建议填写。
- 模型参数：Demo 阶段可显示，产品化后隐藏到服务端配置。
- 运行模式：mock / 真实模型。

验收标准：

- 用户可以完成一次任务提交。
- 提交后生成唯一任务 ID。
- 前端能展示任务状态。

### 7.2 任务状态页

优先级：P0

状态包括：

- 已提交。
- 生成蓝图中。
- 生成 HTML 中。
- 渲染原型中。
- 裁切规划中。
- 生图中。
- 拼接中。
- 已完成。
- 失败。

验收标准：

- 用户可以看到当前任务进度。
- 失败时展示失败环节和可读错误信息。

### 7.3 结果预览页

优先级：P0

展示内容：

- 最终 H5 长图。
- 完整原型图。
- 原型切片列表。
- 裁切点与段数。
- 可选：蓝图 JSON、HTML 原型、debug 信息。

验收标准：

- 用户可以预览最终图。
- 用户可以下载最终图。
- 内部用户可以查看中间产物定位问题。

### 7.4 原型生成能力

优先级：P0

要求：

- 根据 KV、文案和知识库生成结构蓝图。
- 根据蓝图生成完整 HTML wireframe。
- 输出内容符合原型规范。

验收标准：

- HTML 可被 ImageService 渲染。
- 页面结构包含主要文案和模块。
- 不出现被禁止的生产术语。

### 7.5 智能裁切能力

优先级：P0

要求：

- 根据完整原型图自动规划 1-N 段。
- 输出切点、段数和切点依据。
- 支持 max_segment_height 参数。

验收标准：

- 不按固定三段硬切。
- 常见长页可以避开主体模块进行裁切。
- 输出切片可进入 ComfyUI 生图链路。

### 7.6 分段生图与拼接

优先级：P0

要求：

- 支持 1-N 张原型切片输入。
- 每段使用 KV 和对应原型切片。
- 第 2 段起支持 continuation band。
- 输出最终长图。

验收标准：

- mock 模式可以稳定跑通。
- 真实模型模式可以完成至少一组完整商单样例。
- 最终图尺寸和段数符合预期。

### 7.7 自动质检

优先级：P1

质检项：

- HTML 是否完整。
- 是否含外链资源。
- 是否含禁用生产术语。
- 原型尺寸是否符合要求。
- 裁切点是否落在高信息区域。
- 最终图是否存在明显接缝。
- OCR 识别文字是否严重异常。

验收标准：

- 系统可以输出质检结果。
- 失败时可以定位到问题环节。

### 7.8 人工审核与重跑

优先级：P1

能力：

- 支持只重跑原型。
- 支持只重跑裁切。
- 支持只重跑某一段生图。
- 支持重跑整条链路。

验收标准：

- 用户不需要每次从头重新生成。
- 中间产物可以复用。

## 8. 数据与产物

每个任务需要保存：

- `job_id`
- 输入 KV
- 活动文案
- 技术要求
- 蓝图 JSON
- HTML 原型
- 完整原型图
- 裁切计划
- 原型切片
- 生图参数
- 最终长图
- 运行日志
- 错误信息
- 成本和耗时记录

产品化阶段建议 Dify 不再传递大段 base64，而是由 ImageService 或后端保存图片，前端和工作流只传 URL、job ID 和元数据。

## 9. 非功能需求

### 9.1 性能

- Demo 阶段允许单任务耗时较长。
- 产品化阶段需要记录每个环节耗时。
- 生图阶段需要支持超时和失败重试。

### 9.2 稳定性

- 每个任务应有明确状态。
- 第三方模型失败时不应导致任务信息丢失。
- 中间产物应可追溯。

### 9.3 安全

- API key 不得写入文档、前端代码或导入文件。
- 产品化阶段密钥应由服务端环境变量或密钥管理系统托管。
- 内部素材应避免暴露到不可控公网地址。

### 9.4 可观测性

- 记录每次运行的模型、参数、耗时、成本和错误。
- 保存关键中间产物，方便复盘。

## 10. 成功指标

Demo 阶段：

- Dify Cloud 完成一次 mock 端到端实跑。
- 至少完成一组真实商单样例的完整链路。
- 能展示输入、原型、切片、ComfyUI 链路和最终图。

产品化验证阶段：

- 平均生成耗时。
- 单次生成成本。
- 首次可用率。
- 人工修改次数。
- 接缝失败率。
- 原型结构命中率。
- 文案/OCR 可读率。

## 11. 版本规划

### v0.1：Demo 工作流版

- Dify 工作流。
- ImageService 渲染和裁切。
- ComfyUI N 段 mock。
- 结果展示。

### v0.2：真实模型验证版

- 接入真实 Image2。
- 记录成本和耗时。
- 保存逐段生成结果。
- 补充基础质检。

### v0.3：内部工具版

- 自研前端提交页。
- 后端任务状态管理。
- 结果预览与下载。
- 中间产物管理。
- 基础重跑能力。

### v1.0：产品化试用版

- 多活动类型支持。
- 稳定对象存储。
- 密钥管理。
- 自动质检与失败重试。
- 真实业务样例库和效果评估指标。

## 12. 当前风险

- Dify Cloud 与本机 ImageService 需要公网地址打通。
- Dify 传递大图 base64 可能触发变量或请求体上限。
- 真实 Image2 的 N 段成功率、耗时和成本尚未系统验证。
- 生图模型对复杂文字和严格版式仍不稳定。
- continuation band 对复杂画面不一定总是无缝。
- 当前逐段生成图尚未形成稳定对外返回契约。

## 13. 待确认问题

1. 产品化后是保留 Dify 作为后台编排，还是迁移为自研后端 workflow？
2. 前端首版是否只做内部工具，不做多用户权限？
3. 任务产物使用本地存储、对象存储，还是公司内部素材系统？
4. 真实商单验收标准由谁定义：运营、设计、项目负责人，还是三方共同确认？
5. 文字处理是否进入 v0.2，还是延后到 v0.3？
6. 是否需要支持 828px 原始积木规范，还是继续统一到 750px 链路？

## 14. 一句话总结

商单 H5 自动生成工具的目标，是把商单长图从“人工逐步制作”升级为“AI 规划结构、程序控制裁切、模型分段生成、系统自动拼接”的可追踪生产链路。
