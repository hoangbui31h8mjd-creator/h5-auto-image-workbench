# H5 Agent 生产工作台 PRD v0.1

更新时间：2026-07-09

## 1. 项目定位

H5 Agent 生产工作台是一个面向内部 H5 视觉生产流程的本地工具。

它不是单纯的生图工具，也不是完整的 Figma 类自由设计软件，而是一个基于内部设计规范和组件体系的 H5 自动生产工作台。

第一阶段目标是让单个使用者可以在本地完成：

```text
输入需求
-> Agent 补全信息
-> 生成规范化 H5 原型
-> 人在原型阶段修改
-> 根据确认后的原型生成 H5 长图
-> 导出结果
```

后续阶段再扩展生成后图片修改、局部重绘、素材替换和团队内部使用能力。

## 2. 背景与问题

当前项目已经具备一定的 H5 Agent 流程能力，可以基于用户需求生成一张 H5 长图。但现阶段的主要问题是：

- 生成结果偏“一次性出图”，后续修改能力弱。
- 原型阶段没有形成稳定的可编辑工作台，用户难以在生图前调整结构。
- 生成结果缺少结构化数据承载，后续很难做组件替换、局部重生成和版本管理。
- 现有能力依赖本地 runner、规则文档和人工操作，尚未形成完整产品体验。
- 内部已有设计规范和组件经验，但还没有被产品化为可复用的 H5 组件库。

因此，新项目需要把现有流程升级为：

```text
先生成可编辑原型，再基于原型生产最终视觉。
```

## 3. 产品目标

### 3.1 第一阶段目标

第一阶段服务对象是产品负责人本人，本地使用，不要求多人协作。

必须完成最小闭环：

```text
需求输入
-> 生成原型
-> 人改原型
-> 生成长图
-> 导出
```

### 3.2 第二阶段目标

第二阶段服务设计团队内部试用，但暂不做多人同时协作。

重点提升：

- 原型编辑效率
- 组件库覆盖率
- 生图质量
- 长图拼接质量
- 生成后图片修改能力
- 任务历史和版本回溯能力

### 3.3 暂不追求的目标

第一阶段不做以下能力：

- 多人实时协同编辑
- 完整账号体系
- 权限审批流
- 商业化计费
- 面向外部客户的公网产品
- 完整自由画布设计能力
- 复杂模板市场

## 4. 未来产品预期

### 4.1 长期产品愿景

长期来看，本项目希望从“本地 H5 生图工具”演进为“内部 H5 生产工作台”。

它的核心价值不是替代设计师，而是把重复性、规则性、结构化的 H5 生产流程产品化，让设计师把更多时间放在判断、审美、创意和最终质量把控上。

未来理想状态：

```text
运营或设计师输入需求
-> 系统根据内部规范生成 H5 原型
-> 人在原型阶段确认信息结构和组件
-> 系统自动生成视觉长图
-> 人对局部不满意的区域进行修改
-> 系统保存版本和生产记录
-> 导出长图、切片包、HTML 或交付包
```

### 4.2 产品形态预期

未来产品形态可以分三步演进。

第一步是本地个人工具：

```text
单人本地运行
主要用于验证流程
依赖本地 runner 和 API key
重点解决从需求到可编辑原型再到长图导出的闭环
```

第二步是设计团队内部工具：

```text
设计师和运营同学可以使用
不要求多人实时协作
支持项目历史、版本保存、组件库、素材库和局部修改
重点提升真实生产效率
```

第三步是内部 H5 生产平台：

```text
沉淀公司内部 H5 生产标准
支持多类型活动页
支持不同品牌或业务线规范
支持稳定的生图链路和质量评估
支持批量生产、审核、归档和复用
```

### 4.3 能力演进预期

未来能力可以按以下方向扩展。

#### 4.3.1 从“一张图”到“H5 Artifact”

当前结果是一张 H5 长图。

未来结果应该是一个完整 H5 Artifact：

```text
需求
原型 JSON
组件结构
文案
素材
prompt
切片
最终长图
版本记录
导出文件
```

这样后续才能支持编辑、复用、局部重生成和质量追踪。

#### 4.3.2 从“生图后修改”前移到“原型阶段修改”

未来不应该等图片生成完才修改。

更合理的流程是：

```text
先修改原型
再生成视觉
最后只做局部微调
```

原型阶段需要支持：

- 改文案
- 换组件
- 调整模块顺序
- 换主题
- 替换图片占位
- 重新生成某个模块
- 重新生成整版原型

这会显著降低修改成本。

#### 4.3.3 从“固定流程”到“组件化生产”

未来 H5 不应该只靠一整段 prompt 生成，而应该由组件组成。

示例：

```text
Hero / KV
活动标题
奖励展示
角色或商品展示
规则说明
CTA
底部信息
```

每个组件都有自己的适用场景、参数、样式规则和生图规则。用户可以像替换积木一样替换组件，而不是每次都从零生成。

#### 4.3.4 从“人工经验”到“内部规范产品化”

内部已有设计规范未来应该被系统读取和执行。

规范不只是一份文档，而应该影响：

- 原型结构
- 组件选择
- 字体大小
- 色彩倾向
- 图片风格
- 文案长度
- 模块顺序
- CTA 位置
- 导出规格

目标是让 Agent 在生成阶段就尽量符合规范，而不是生成后靠人工大修。

#### 4.3.5 从“整张重做”到“局部可控修改”

未来生成后图片修改应该优先支持模块级和区域级修改。

例如：

- 只重生成 Hero 背景
- 只替换奖励区素材
- 只修改按钮样式
- 只调整角色图位置
- 保留其他区域不变

这要求第一阶段就保存模块、切片和 prompt 记录。

#### 4.3.6 从“单次任务”到“可追溯生产记录”

未来每一次生成都应该可追溯：

- 用户输入了什么需求
- Agent 问了什么问题
- 用户补了什么信息
- 生成了哪个原型版本
- 用了哪些组件
- 用了哪些 prompt
- 哪些模块重生成过
- 最终导出了哪个版本

这对团队复盘、质量改进和规范沉淀非常重要。

### 4.4 未来用户体验预期

未来工作台可以采用类似设计工具的布局，但不做完整自由画布。

建议形态：

```text
左侧：项目 / 历史 / 组件库 / 模板
中间：H5 原型预览或长图预览
右侧：当前模块属性、文案、图片、主题、生成参数
底部或侧栏：Agent 对话、补问、生成日志
```

用户体验重点：

- 用户不需要理解底层 runner。
- 用户不需要手写复杂 prompt。
- 用户可以在原型阶段看懂结构并修改。
- 用户可以知道当前缺什么信息。
- 用户可以知道失败原因。
- 用户可以回到历史版本。
- 用户可以导出可交付结果。

### 4.5 与 Open Design 的关系预期

Open Design 可以作为长期参考对象，但本项目不应该变成 Open Design 的简单换皮。

可借鉴：

- 本地优先工作台形态
- Agent 调度方式
- Codex / Claude CLI 接入方式
- skills 机制
- design-systems 机制
- artifact 预览和导出方式
- 插件化扩展思路

需要替换成自有能力：

- 通用设计 artifact -> H5 Artifact
- 通用 design systems -> 内部 H5 设计规范
- 通用 skills -> H5 原型、生图、切片、拼接、导出 skills
- 通用工作台 -> H5 生产工作台
- 通用插件 -> 内部组件库、模板库和生图工作流

长期策略是：

```text
借 Open Design 的架构思想
但沉淀自己的 H5 生产标准、组件库和 artifact 协议
```

### 4.6 未来成功状态

如果产品方向成立，未来应该达到以下状态：

- 设计师可以用它快速生成符合内部规范的 H5 初稿。
- 运营可以用自然语言描述活动需求，不需要写复杂 prompt。
- 原型阶段能解决大部分结构和内容问题。
- 生图阶段主要负责视觉完成度，而不是重新理解需求。
- 生成后的图片可以局部修改，而不是整张重来。
- 每个 H5 都有结构化记录，可以复用、回溯和沉淀。
- 组件库和规范会随着真实项目持续变强。
- 团队能明显减少重复排版、切图、拼接和基础出稿时间。

## 5. 用户与场景

### 5.1 第一阶段用户

第一阶段用户是产品负责人本人。

使用场景：

- 验证 H5 自动生产流程是否成立。
- 通过真实需求测试原型生成、生图和导出质量。
- 沉淀内部 H5 组件、规范和流程。
- 判断该工具是否值得推进给设计团队试用。

### 5.2 第二阶段用户

第二阶段用户是内部设计师和运营相关同学。

使用场景：

- 运营提出活动需求后，快速生成 H5 初稿。
- 设计师在原型阶段调整结构和组件。
- 设计师确认原型后，一键生成视觉长图。
- 对生成结果进行局部修改和导出。

## 6. 核心产品流程

### 6.1 MVP 主流程

```text
1. 用户输入 H5 需求
2. Agent 检查 KV、文案、语言、活动类型、尺寸、素材等关键信息
3. 缺信息时先追问，不直接生成
4. 信息完整后生成结构化 H5 原型
5. 用户在原型阶段修改文案、组件、顺序、主题和图片
6. 用户确认原型
7. 系统基于最终原型生成生图提示词和生产任务
8. 生图、切片、拼接成长图
9. 用户预览结果
10. 导出 PNG / WebP / 项目包
```

### 6.2 下一阶段流程

```text
1. 用户选择生成后的长图
2. 选择需要修改的区域或模块
3. 输入修改要求
4. 系统进行局部重生成或素材替换
5. 更新长图版本
6. 用户导出新版本
```

## 7. 功能范围

### 7.1 需求输入模块

用户可以输入自然语言需求，例如：

```text
帮我做一张巴西葡语的游戏活动 H5，主题是充值返利，风格要有冲击力，突出奖励。
```

系统需要识别并检查：

- H5 类型
- 目标语言
- 活动主题
- KV 信息
- 标题和正文
- 奖励信息
- CTA 文案
- 品牌或项目规范
- 必要素材
- 输出尺寸

默认语言为巴西葡语 `pt-BR`，除非用户明确指定其他语言。

### 7.2 信息补问模块

当关键信息缺失时，Agent 必须先补问。

示例：

```text
当前缺少活动主标题、奖励内容和 CTA 文案。请补充这三项后，我再生成 H5 原型。
```

补问原则：

- 只问当前生成必须的信息。
- 一次最多问 3-5 个关键问题。
- 能根据内部规范推断的内容，不打扰用户。
- 不能在 KV 或核心内容缺失时直接生图。

### 7.3 原型生成模块

系统根据需求生成一个规范化 H5 原型。

原型不是最终图片，而是一个可编辑结构，包含：

- 页面模块
- 组件类型
- 文案
- 图片占位
- 布局顺序
- 主题风格
- 组件参数
- 生图指令草稿

原型需要符合内部设计规范。

### 7.4 原型编辑模块

第一阶段希望支持以下编辑能力：

- 修改文案
- 替换组件
- 调整模块顺序
- 修改颜色主题
- 替换图片或图片占位
- 局部重生成某个模块
- 重新生成整版原型

原型编辑优先采用组件化方式，不做完整自由画布。

示例：

```text
Hero 组件：hero_kv_v1 -> hero_kv_v3
奖励组件：reward_grid_v1 -> reward_cards_v2
CTA 组件：bottom_button_v1 -> sticky_cta_v1
```

### 7.5 组件库模块

系统需要有一套内部 H5 组件库。

第一阶段建议至少包含：

- Hero / KV 组件
- 活动标题组件
- 奖励展示组件
- 商品或角色展示组件
- 规则说明组件
- CTA 按钮组件
- 底部信息组件
- 分割装饰组件

每个组件需要包含：

- 组件名称
- 适用场景
- 可编辑字段
- 视觉规则
- 默认尺寸
- 生图提示词规则
- 是否支持局部重生成

### 7.6 设计规范模块

系统需要读取并应用内部设计规范。

规范可以包括：

- 字体规则
- 色彩规则
- 版式规则
- 按钮规则
- 活动页结构规则
- 不同 H5 类型的默认模块顺序
- 不同品牌或项目的视觉约束

目标是让 Agent 生成原型时先遵守规范，而不是生成后再人工大修。

### 7.7 生图生产模块

用户确认原型后，系统进入生图生产。

生图生产需要基于原型结构，而不是只基于用户一句话。

输入应包含：

- 原型 JSON
- 组件信息
- 文案
- 图片占位
- 内部设计规范
- 目标风格
- 输出尺寸
- 生图模型参数

输出应包含：

- 每个模块或切片的图片
- 拼接后的 H5 长图
- 生成日志
- prompt 记录
- 失败信息

### 7.8 导出模块

第一阶段至少支持：

- 导出 H5 长图 PNG
- 导出 H5 长图 WebP
- 导出项目包

项目包建议包含：

```text
final.png
prototype.json
sections/
prompts/
metadata.json
```

第二阶段可增加：

- HTML 导出
- 切片包导出
- 设计交付包导出

### 7.9 生成后图片修改模块

这是 MVP 后的下一阶段能力。

支持：

- 选择某个模块重生成
- 选择局部区域重绘
- 替换某张素材
- 修改背景
- 修改按钮或文字区域
- 保留其他区域不变
- 生成新版本

该模块依赖前面的结构化原型和模块切片数据。如果第一阶段只保存一张最终长图，后续图片修改会非常受限。

## 8. H5 Artifact 数据结构

本项目的核心不是只生成图片，而是生成一个 H5 Artifact。

一个 H5 Artifact 至少包含：

```text
1. 用户原始需求
2. 信息补问记录
3. 原型 JSON
4. 组件列表
5. 文案内容
6. 图片素材或图片占位
7. 内部设计规范引用
8. 生图 prompt
9. 生图参数
10. 切片结果
11. 最终长图
12. 版本记录
```

建议原型 JSON 结构示例：

```json
{
  "artifactType": "h5_campaign",
  "language": "pt-BR",
  "pageSize": {
    "width": 750,
    "height": 3000
  },
  "theme": {
    "style": "game_campaign",
    "brand": "internal_default"
  },
  "sections": [
    {
      "id": "hero",
      "component": "hero_kv_v1",
      "props": {
        "headline": "Título da campanha",
        "subtitle": "Subtítulo da campanha",
        "imageSlot": "hero_character"
      }
    },
    {
      "id": "rewards",
      "component": "reward_grid_v1",
      "props": {
        "items": []
      }
    },
    {
      "id": "cta",
      "component": "bottom_cta_v1",
      "props": {
        "buttonText": "Participar agora"
      }
    }
  ]
}
```

## 9. 开源能力复用建议

### 9.1 Open Design

可参考能力：

- Agent 工作台结构
- Codex / Claude CLI 接入方式
- skills 机制
- design-systems 机制
- artifact 预览与导出思路
- 本地优先产品形态

不建议全盘照搬。

本项目应该将 Open Design 的通用设计工作台思想，收敛为 H5 生产工作台。

### 9.2 Codex

Codex 适合承担：

- 理解用户需求
- 检查缺失信息
- 调用本地 runner
- 生成结构化原型
- 生成生图 prompt
- 执行局部修改任务
- 总结错误和生产日志

Codex 不应该承担所有图像生产职责。高频生图仍应由专门的图像模型或工作流执行。

### 9.3 ComfyUI 或现有生图链路

适合承担：

- 模块图生成
- 局部重绘
- 风格化
- 放大
- 图像后处理

如果当前已有生图 runner，可以优先沿用，不必第一阶段重建完整 ComfyUI 工作流。

### 9.4 Web 编辑器能力

可以参考 GrapesJS、tldraw 等开源项目，但第一阶段不建议做完整自由画布。

更建议先做：

```text
组件化编辑器
```

也就是页面由一组模块组成，用户通过属性面板修改模块，而不是自由拖动每一个像素。

## 10. 技术形态建议

### 10.1 第一阶段形态

第一阶段建议做成本地 Web 工具。

```text
本地前端页面
本地后端服务
Codex / runner / 生图接口
本地文件存储
```

优点：

- 启动快
- 不需要账号体系
- 不需要多人协作
- 便于调试本地 runner
- 便于复用当前项目文件

### 10.2 未来形态

未来可以演进为：

- 内部 Web 工具
- 桌面应用
- 内网部署服务

但这些不应该进入第一阶段核心范围。

## 11. 验收标准

### 11.1 MVP 验收标准

完成以下能力即可视为 MVP 成立：

- 用户可以输入一个 H5 需求。
- 系统能识别关键信息是否缺失。
- 缺失核心信息时，系统会先补问。
- 信息完整后，系统生成结构化 H5 原型。
- 用户可以在原型阶段修改文案。
- 用户可以替换至少 3 类核心组件。
- 用户可以调整模块顺序。
- 用户可以确认原型并触发生图。
- 系统可以生成 H5 长图。
- 用户可以导出最终长图。
- 每次生成都保存原型 JSON、prompt 和最终结果。

### 11.2 质量验收标准

建议用 10 个真实 H5 需求进行测试。

通过标准：

- 8/10 个需求可以生成完整原型。
- 8/10 个原型符合基础结构规范。
- 7/10 个生成结果不需要重新走完整流程即可修改到可用。
- 生成过程中的关键文件可追溯。
- 用户能明确知道失败原因和下一步动作。

## 13. 风险与对策

### 13.1 风险：一开始做得太像 Figma

如果第一阶段追求自由画布、任意拖拽、精细排版，会显著拉长周期。

对策：

第一阶段坚持组件化编辑，不做完整自由画布。

### 13.2 风险：只生成图片，缺少结构数据

如果最终结果只有一张 PNG，后续很难支持原型修改、局部重生成和版本管理。

对策：

从第一阶段开始保存 H5 Artifact，包括原型 JSON、组件列表、prompt 和切片信息。

### 13.3 风险：生图结果不可控

模型输出可能不稳定，导致同一规范下视觉质量波动。

对策：

通过组件库、设计规范、prompt 模板和固定生产流程降低随机性。

### 13.4 风险：Open Design 魔改成本过高

Open Design 是通用设计工作台，直接深度魔改可能带来大量无关复杂度。

对策：

优先借鉴其架构思想和局部能力，不把第一阶段目标定义为完整 fork 改造。

### 13.5 风险：本地 runner 和 API key 配置复杂

第一阶段仍可能依赖本地环境，影响使用体验。

对策：

先为个人本地使用做清晰启动脚本和环境检查；团队阶段再考虑一键应用或内部服务。

## 15. 第一阶段建议任务拆分

### 任务 1：定义 H5 Artifact 协议

明确 H5 的结构化数据格式，包括页面、模块、组件、文案、素材、prompt 和版本。

### 任务 2：整理第一批组件库

从内部已有设计规范中抽取最常用的 H5 模块。

建议先做 6-8 个组件，不追求一次覆盖所有类型。

### 任务 3：搭建本地工作台骨架

包含：

- 需求输入区
- 原型预览区
- 属性编辑区
- 组件替换区
- 生图执行区
- 输出结果区

### 任务 4：接入 Codex Agent 流程

让 Codex 负责：

- 读需求
- 检查信息
- 追问缺失项
- 生成原型 JSON
- 生成生图任务

### 任务 5：接入生图和导出

打通：

```text
原型 JSON -> 生图 prompt -> 图片生成 -> 长图拼接 -> 导出
```

### 任务 6：用真实案例验证

用至少 10 个真实 H5 需求测试。

记录：

- 原型是否合理
- 哪些组件不够用
- 哪些规范没有被遵守
- 生图哪里不可控
- 哪些修改最常发生

## 16. 当前结论

这个产品方向成立，但第一阶段必须收敛。

最重要的不是先做一个大而全的设计平台，而是先完成一个可靠的闭环：

```text
规范化原型
-> 人可编辑
-> 基于原型生图
-> 可导出
```

只要这个闭环成立，后续才有基础继续做生成后图片修改、团队内部使用和更完整的 H5 生产工作台。
