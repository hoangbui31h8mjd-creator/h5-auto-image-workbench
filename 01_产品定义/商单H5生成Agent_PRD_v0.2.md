## 0. 产品推导逻辑

本节用于说明：为什么这个产品不应该被定义成一个简单的“AI 生图工作流”，而应该被定义成一个面向商单 H5 生产的垂直 Agent。

从最底层看，商单 H5 长图生产不是单纯追求“画面好看”，而是要把一组业务输入稳定转化为一张可交付、可解释、可复盘的移动端长图。

因此，一个可用系统至少要解决 7 件事：

```text
业务意图
-> 页面结构
-> 低保真原型
-> 可控裁切
-> 分段视觉生成
-> 无缝拼接
-> 结果验收与归档
```

由此倒推：

- 只接生图模型不够，因为模型会自由发挥，缺少页面结构约束。
- 只做 HTML 原型不够，因为最终视觉仍需要模型生成。
- 只固定三段切图不够，因为真实 H5 长度不固定，硬切会切断主体模块。
- 只做拼接不够，因为分段图之间需要风格一致和视觉续接。
- 只靠 Prompt 不够，因为公司业务经验、品牌规范和历史案例需要长期沉淀。

所以这个产品的合理形态不是“一个节点”或“一个模型调用”，而是一个小型 Agent 系统：

```text
公司知识库：提供规则、案例和品牌经验
Skill / Prompt：把经验转成稳定执行规范
LLM：理解需求并生成页面结构
ImageService：负责渲染、裁切、拼接等确定性处理
Image2：负责最终视觉生成
任务系统：保存中间产物、状态和结果
```

这也是本 PRD 后续功能拆分的依据。

## 1. 项目摘要

### 1.1 问题定义

商单 H5 长图制作依赖人工理解需求、搭建原型、切图、生图、拼接和反复修改。活动结构高度重复，但每次仍需要大量人工判断，尤其在海外活动、游戏活动、任务奖励类页面中，交付周期紧、沟通成本高。

当前 AI 生图工具可以生成视觉图，但缺少公司知识库、页面结构约束、裁切策略和长图拼接逻辑，因此无法直接稳定承担商单 H5 生产任务。

### 1.2 解决方案

建设一个“商单 H5 生成 Agent”：用户输入 KV 图和内容信息后，系统自动调用公司知识库与规则 Skill，生成 HTML 原型、规划模块边界切片、调用 Image2 分段生成，并将分段结果拼接成完整 H5 长图。语言、尺寸、组件、切图、接龙等默认规则由 Skill 承担；用户只有在有特殊要求时才需要补充。

当前 Demo 继续允许 Dify 负责编排、ComfyUI 负责生图链路验证；产品化阶段逐步迁移为自研后端 + 独立知识库 + 前端任务系统。

### 1.3 成功标准

Demo 阶段成功标准：

- 至少 1 个真实活动样例完成 `KV + 文案 + 要求 -> 原型 -> 切片 -> 生图 -> 拼接 -> 输出` 的端到端链路。
- HTML 原型无需人工重写即可被 ImageService 渲染，渲染成功率达到 90% 以上。
- 裁切点不切断主要标题、按钮、奖励卡、任务卡、排行榜、FAQ 等主体模块。
- 最终长图能够回传并保存在项目结果目录中，包含完整原型图、切片、最终图和运行摘要。
- 单次任务失败时能定位失败环节，不需要从零猜测问题。

内部工具阶段成功标准：

- 单任务可以通过 Web 页面提交并获得任务 ID。
- 任务状态可追踪，至少覆盖“排队、原型生成、裁切、生图、拼接、完成、失败”。
- 中间产物可查看、下载和复用。
- 公司知识库能够提供规则、案例和风格参考，不依赖单次人工口述。
- API key 不出现在前端、文档、工作流导入文件或结果产物中。

## 2. 产品定义

### 2.1 产品名称

商单 H5 生成 Agent。

### 2.2 产品定位

面向公司内部商单、运营和设计场景的 AI 长图生产助手。

它不是一个泛用绘图工具，而是一个受控生产系统：

```text
输入：KV 图 + 内容信息 + 可选特殊要求 + 公司知识库
输出：H5 HTML 原型 + 切片方案 + 分段生成图 + 最终 H5 长图
```

### 2.3 当前形态

当前仍是 Demo / 验证形态：

- Dify：负责输入、知识库检索、LLM 原型生成、流程编排和结果展示。
- ImageService：负责 HTML 渲染、裁切规划、切片、文件保存、调用 ComfyUI。
- ComfyUI：负责 Image2 分段生成、续画、羽化拼接。
- OpenRouter / Image2：负责最终视觉生成。

### 2.4 目标形态

产品化后不必须依赖 Dify 和 ComfyUI 页面操作。推荐目标形态是：

```text
前端任务页
-> H5 Agent 后端
-> 独立公司知识库服务
-> HTML 渲染与裁切服务
-> Image2 生图服务
-> 拼接与质检服务
-> 结果归档与展示
```

Dify 和 ComfyUI 在当前阶段是验证工具；产品本质是“带知识库和图像处理能力的 H5 生成 Agent”。

## 3. 目标用户

### 3.1 个人使用者 / 项目推进人

典型用户：当前项目负责人。

核心诉求：

- 快速验证一条商单 H5 自动生成链路。
- 能把 KV 和内容信息丢进去，获得可展示的初步结果；特殊要求可选。
- 能沉淀案例、文档和工作流，方便给老板或同事汇报。
- 允许少量手动配置和调试，但希望每次结果可追踪。

### 3.2 运营 / 商单执行同学

核心诉求：

- 提交活动需求，快速看到第一版 H5 长图。
- 不需要理解 Dify、ComfyUI、API endpoint、图片 base64。
- 能清楚看到任务是否还在跑、哪里失败、结果在哪里下载。

### 3.3 设计师 / 创意同学

核心诉求：

- 减少重复搭建任务页、奖励页、榜单页、规则页的工作。
- 获取一个结构正确、风格接近的初稿。
- 能查看原型、切片和分段图，判断哪里需要精修。

### 3.4 项目负责人 / 管理者

核心诉求：

- 判断 AI 自动化是否能降低商单物料生产成本。
- 评估生成质量、稳定性、耗时和成本。
- 判断后续是否值得投入内部工具化或平台化。

## 4. 用户故事与验收标准

### 4.1 提交任务

用户故事：

作为运营同学，我希望上传 KV、填写活动内容或页面板块，以便系统自动生成一张 H5 长图初稿。

验收标准：

- 支持上传 jpg/png 格式 KV。
- 支持填写活动内容、页面板块、奖励、规则和目标用户。
- 支持可选特殊要求：其他语言、固定尺寸、特殊调性、禁用项和业务约束。
- 默认语言为巴西葡语 `pt-BR`，用户明确指定其他语言时才覆盖。
- 提交后生成唯一 `job_id`。
- 用户不需要填写密钥、endpoint 和本地路径。

### 4.2 生成 HTML 原型

用户故事：

作为设计师，我希望系统先生成低保真 HTML 原型，以便我确认页面结构而不是直接看模型自由发挥的成品图。

验收标准：

- HTML 是完整自包含文档，可被浏览器或 ImageService 渲染。
- 原型宽度默认 828px，跟随积木/Figma 组件规范；如用户指定其他宽度，必须全链路统一换算。
- 原型是黑白灰 wireframe，不是最终视觉稿。
- 原型包含真实模块名、任务、奖励、按钮、FAQ、规则等业务信息。
- 原型不得显示“安全区、40px、80px、切口、接缝、续画”等生产术语。
- KV 不直接嵌入原型作为成品头图，除非用户明确要求“以 KV 作为首屏实际素材”。

### 4.3 自动切片

用户故事：

作为系统使用者，我希望系统按模块边界自动切片，以免把主体内容切断导致后续生图失败。

验收标准：

- 切片数量支持 1-N，不固定三段。
- 单张原型切片高度预算为 `当前原型图宽度 × 3 - 80`，确保追加 80px 参考带后仍不超过 1:3。
- 如果理论切点落在模块中间，系统应向上寻找最近模块结束位置或低信息区域。
- 不切断主要标题、任务卡、奖励卡、按钮、表格、视频框、排行榜、FAQ 条目。
- 输出裁切计划，包括切点、段数和切点依据。

### 4.4 分段生图

用户故事：

作为项目负责人，我希望系统能根据 KV 和原型切片分段生成长图，以突破单张生图尺寸限制。

验收标准：

- 第 1 段使用 KV + 第 1 张原型切片生成。
- 第 2 段起使用 KV + 当前原型切片 + 上一段底部参考带生成。
- 原型切片作为布局主参考，KV 作为风格主参考。
- Image2 请求应传入合理尺寸，避免模型按默认比例生成后被强行拉伸。
- 每段生成结果应可保存，至少在调试模式下可查看。

### 4.5 拼接输出

用户故事：

作为业务方，我希望最终看到完整长图，而不是多个分段结果。

验收标准：

- 最终图按段顺序拼接。
- 相邻段使用上一段底部参考带作为对齐基础。
- 重叠区做羽化过渡，减少硬接缝。
- 最终图、完整原型图、原型切片、运行摘要保存到结果目录。
- Dify 或前端能直接展示最终图，不要求用户复制本地路径再打开。

### 4.6 知识库参考

用户故事：

作为公司内部使用者，我希望系统参考公司知识库生成内容，以便输出符合公司业务经验和品牌规范。

验收标准：

- 知识库支持文字规则、案例说明、图片案例和结构化标签。
- 任务执行前能检索与当前 KV、活动类型、语言、业务场景相关的资料。
- 模型只接收当前任务必要的知识片段，不暴露完整公司资料。
- 敏感资料不直接传给外部模型，或经过脱敏与摘要处理后再使用。

## 5. 非目标

当前阶段明确不做：

- 不承诺最终图达到设计师精修上线水平。
- 不做真实 H5 前端交互页面发布，只输出长图和中间产物。
- 不做复杂多人权限、审批流、计费系统。
- 不做全公司通用设计平台。
- 不承诺所有活动类型一次生成稳定成功。
- 不把 API key、公司内部资料或敏感素材写入工作流文件。

## 7. AI 系统需求

### 7.1 AI 角色拆分

#### 7.1.1 原型生成 LLM

职责：

- 理解 KV、内容信息、可选特殊要求和知识库。
- 输出页面蓝图 JSON。
- 输出 HTML wireframe。

不得做：

- 不直接生成最终视觉稿。
- 不把 KV 像素嵌入原型。
- 不输出生产术语到可见页面。

#### 7.1.2 生图执行分析 LLM

职责：

- 根据 KV 和已经切好的原型切片分析每段视觉角色。
- 生成每段 Image2 提示词。
- 说明参考带是上一段延续，不是新模块。

不得做：

- 不重新规划 HTML 页面结构。
- 不改变模块顺序。

#### 7.1.3 Image2

职责：

- 根据 KV 风格和原型切片版式生成分段视觉图。
- 保持整体风格一致。
- 尽量保留模块空间、卡片数量、按钮位置和信息层级。

限制：

- 对精确文字和严格 UI 还不稳定。
- 对超长图需要分段生成。
- 需要明确尺寸/比例参数，避免拉伸。

### 7.2 提示词原则

Prompt 不应只写“生成一张好看的 H5 图”，而应包含施工规范：

- 原型切片是布局主参考。
- KV 是风格主参考。
- 保持模块顺序和相对位置。
- 灰色占位转换为主题化图标、道具、图片面板或纹理区域。
- 可见文案使用目标语言；默认使用巴西葡语 `pt-BR`，用户明确指定其他语言时才覆盖。
- 不生成安全区、切口、接缝、40px 等生产标记。
- 第 2 段起顶部参考带来自上一段真实画面，只用于续接。

### 7.3 评估策略

自动评估：

- HTML 是否完整可渲染。
- 原型是否包含主要业务模块。
- 原型是否出现禁用词。
- 切点是否落在模块边界或低信息区。
- 切片高度是否超出模型约束。
- 最终图尺寸是否符合预期。
- 接缝处是否存在明显白线、灰条、重复标题或错位。

人工评估：

- 风格是否符合 KV。
- 页面结构是否符合活动目标。
- 视觉是否达到可讨论初稿水平。
- 是否需要设计师精修。

## 8. 知识库需求

### 8.1 为什么需要独立知识库

如果知识只放在 Dify 里，Demo 阶段方便，但未来产品化会受限：

- 难以脱离 Dify 迁移到自研后端。
- 图文混合案例难以做结构化管理。
- 公司内部资料安全和权限不好统一控制。
- 不利于复用到其他 Agent 或项目。

因此产品化目标是建立独立公司 H5 知识库。

### 8.2 知识库内容形态

知识库不是只存文档，而是“原始文件 + 结构化理解 + 检索索引”。

建议目录或服务形态：

```text
H5_Company_KnowledgeBase/
├── raw_assets/              原始资料：KV、长图、HTML、Excel、PDF、图片案例
├── parsed_data/             OCR、图像描述、结构化标签
├── rules/                   品牌规则、H5 模块规则、Prompt 规则
├── case_index/              案例元数据
├── vector_index/            文本和图片检索索引
└── kb.sqlite / PostgreSQL   结构化数据库
```

### 8.3 图片知识的处理方式

图片不能只作为图片存储，需要补充可检索信息：

- OCR：识别图中文字。
- 视觉描述：颜色、材质、风格、构图。
- 版式结构：头图、任务、奖励、榜单、FAQ、CTA。
- 标签：业务类型、语言、风格、适用场景、禁用项。
- 原图路径：指向文件存储，不把大图塞进模型上下文。

示例数据：

```json
{
  "case_id": "pubg_001",
  "business_type": "game_campaign",
  "language": "pt-BR",
  "style_tags": ["black_gold", "battle", "military", "reward"],
  "modules": ["hero", "intro", "task_list", "reward_grid", "leaderboard", "faq", "cta"],
  "raw_image_path": "raw_assets/pubg_001/final_h5.png",
  "prototype_html_path": "raw_assets/pubg_001/prototype.html",
  "notes": "适合版本挑战、任务奖励、排行榜类活动"
}
```

### 8.4 检索 API

内部产品化建议提供：

- `POST /kb/search_text`：检索文字规则。
- `POST /kb/search_case`：检索相似业务案例。
- `POST /kb/search_image_style`：按 KV 风格查找相似案例。
- `GET /kb/cases/{case_id}`：获取案例详情。
- `POST /kb/ingest`：新增案例并解析入库。

## 9. 技术方案

### 9.1 当前 Demo 架构

```text
Dify
-> ImageService
-> ComfyUI
-> OpenRouter / Image2
-> ImageService 保存结果
-> Dify 展示
```

当前关键参数：

- 原型宽度：默认 828px。
- 切片数量：1-N。
- 单张原型切片高度预算：当前原型图宽度 × 3 - 80。
- 续画/羽化参考带：当前默认 80px，后续可配置。
- ComfyUI 主实例：`http://127.0.0.1:8188`。
- ImageService 本机端口：`http://127.0.0.1:8000`。

### 9.2 产品化推荐架构

```text
Web 前端
  |
H5 Agent API
  |
Job Queue / Worker
  |
+----------------------+----------------------+----------------------+
| Knowledge Service    | Render & Slice       | Image Generation     |
| 规则 / 案例 / 检索    | HTML 渲染 / 裁切       | Image2 分段生成       |
+----------------------+----------------------+----------------------+
  |
Storage / DB
  |
Result Preview / Download
```

### 9.3 后端模块

#### 任务 API

- 创建任务。
- 查询任务状态。
- 获取结果。
- 重跑某个环节。

#### 知识库服务

- 存储公司规则和案例。
- 做文字和图片检索。
- 返回与当前任务相关的少量知识片段。

#### 原型生成服务

- 调 LLM 生成蓝图和 HTML。
- 校验 HTML 是否可渲染。
- 保存 HTML 和蓝图。

#### ImageService

- HTML 渲染。
- 模块边界裁切。
- 切片保存。

#### 生图执行器

- 调 Image2。
- 管理 1-N 段生成。
- 做参考带续接。
- 保存分段图和最终图。

#### 质量检查服务

- 规则质检。
- 图像尺寸质检。
- 接缝质检。
- 文案/OCR 质检。

## 10. 安全与隐私

### 10.1 密钥管理

- API key 不得写入 Dify 导入文件、Markdown 文档、前端代码或日志。
- Demo 阶段可以由用户在运行界面手动填写。
- 产品化阶段必须改为服务端环境变量或密钥管理系统。

### 10.2 公司资料安全

- 公司知识库原文、原图和内部案例默认不直接发给外部模型。
- 检索后只传必要片段、摘要或脱敏后的规则。
- 敏感项目需要支持“只在内网模型处理”。
- 需要记录每次外部模型调用发送了哪些知识片段。

### 10.3 文件访问

- Dify Cloud 不能访问本机 `127.0.0.1`，必须使用可访问的公网 HTTPS 或部署在云端。
- 产品化后不应依赖临时 tunnel。
- 结果图应放入公司可控对象存储或内部文件系统。

## 11. 风险

### 11.1 技术风险

- Dify Cloud 变量、base64、HTTP body 可能存在大小限制。
- Image2 对长图、文字和严格布局仍可能不稳定。
- 多段生成耗时长，单任务可能 5-15 分钟。
- 参考带续接不能保证所有复杂画面都无缝。
- 真实业务案例多样，规则库不足时原型仍会偏。

### 11.2 产品风险

- 如果用户期待“一次生成即可上线”，当前能力会被高估。
- 如果没有中间产物展示，失败排查成本会很高。
- 如果知识库没有结构化，Agent 只会“看过很多资料但不会用”。
- 如果公司资料安全策略不明确，内部推广会受阻。

### 11.3 成本风险

- Image2 多段生成会放大单次成本。
- 重跑某一段、重跑整图都需要成本记录。
- 需要在产品界面展示耗时和成本，避免盲跑。

验收：

目标：

- 多人使用。
- 权限与项目隔离。
- 结果归档。
- 成本统计。
- 基础质检。
- 支持真实业务试点。

验收：
