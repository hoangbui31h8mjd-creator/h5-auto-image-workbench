# H5 原型生成知识库（单文件当前版）

用于 Codex 本地 Agent 根据 KV、内容信息、可选特殊要求和 Skill 规范生成 H5 蓝图与低保真 HTML。本文只约束原型和结构；最终视觉由 Image2 或后续生图模块完成。

# H5 原型生成规则（当前版）

## 阶段边界

本规则只负责蓝图和 HTML wireframe。原型不是最终视觉稿，但用户给的头图/KV 默认就是 H5 第一屏主视觉，必须嵌入 HTML 顶部；后续模块仍使用低保真结构表达，最终美化由 Image2 或后续生图模块完成。

## 当前统一尺寸

- 当前 Demo/链路画布：默认 828px。
- `jumu-components.md` 的源规范是 828px，当前本地 Agent 直接沿用该规范。
- 内容区默认 748px，左右边距 40px。
- 如用户指定其他宽度，必须按 `当前宽度 / 828` 全链路统一换算。

不得在同一条链路里混用 828px 与历史 750px 口径。

## 输出

- 蓝图 JSON：页面模块、文案来源、候选 safe_bands、组件类型、视觉意图。
- 自包含 HTML wireframe。
- 渲染后的完整长原型图。
- 由 ImageService 根据实际高度生成的 1-N 张切片。

## 原型硬规则

1. 使用黑白灰 wireframe，真实模块名和简短业务文案可见。
2. 首屏必须嵌入用户提供的头图/KV；其他模块不嵌入额外真实图片、外链图片、外链字体、脚本或远程资源。
3. 不显示“安全区、40px、80px、切口、接缝、续画、拼接”等生产术语。
4. 不在每段顶部或底部画灰条、白条、虚线安全带。
5. 不固定三段；HTML 先完整生成，裁切服务再决定 1-N 段。
6. 候选 safe_bands 只作为机器字段，尽量位于模块间自然留白或低信息背景。
7. 标题、按钮、人物、商品、视频框、表格、奖励卡和关键文案不得被切断。
8. 页面高度由内容决定，不为凑固定切片数拉伸空白。

## 组件选择规则（由积木 Skill 适配）

- 任务活动：任务列表、进度条、奖励卡、CTA。
- 排名活动：Tab、排行榜、我的位置、奖励说明。
- 内容投稿：Hashtag、视频卡片、头像列表、投票、分享卡片。
- 直播活动：直播预告、倒计时、时间轴、CTA。
- 抽奖活动：翻卡、剩余次数、奖品入口、结果状态；原型只表达结构，不模拟弹层动效。
- 规则与收口：FAQ、规则、免责声明、尾标和最终 CTA。

主标题限制一行；任务副标题按布局限制 2-3 行；视频框保留原始比例，不擅自改横竖版。

## 蓝图字段

```json
{
  "page": {"width": 828, "height_strategy": "content_driven"},
  "modules": [],
  "copy_map": {},
  "safe_bands": [],
  "visual_intent": {},
  "negative_rules": []
}
```

`safe_bands` 是候选区域，不是最终 cuts；最终切点由渲染后的图像分析决定。

## KV 视觉意图（由主题 Skill 适配）

蓝图可以记录但 HTML 不上色：

- 精确主色、辅色、点缀色。
- 风格的材质、边缘、线条粗细和转角特征。
- 顶层、中层、底层背景叙事和模块间过渡方向。
- 每个模块最多三个主要背景元素。

风格观察必须回答：线条粗细是否变化、边缘是硬切还是毛边、转角是圆弧/直角/尖锐楔形。禁止只写“插画风、3D 风、写实风”等空泛标签。

原始 `h5-module-theme.md` 中“完整保留 KV”在当前产品包里默认解释为：KV 作为首屏头图素材进入 HTML 原型，后续模块只延展 KV 的风格、颜色、材质和氛围，不重复复制 KV，也不额外复制真实 logo。

## HTML

- 完整 `<!doctype html>` 到 `</html>`。
- 主容器默认宽 828px，CSS 内联。
- 首屏头图区域使用用户提供的 KV；其他图片/视频区域用灰阶占位，文字使用真实内容。
- 页面为完整连续长页，不提前按切片拆成独立页面。

## 裁切与生图交接

切图模块按 `当前原型图宽度 × 3 - 80` 规划切点。第 2 段起的顶部 80px 参考带由生图链路追加，不由 HTML 原型绘制。

# H5 长图原型案例知识库包（展示版）

用途：给 Dify 知识库上传，用于辅助 LLM 生成“低保真 H5 长图原型”。本文件只保存公开案例的结构观察、模块拆解和来源链接，不复制原始图片资产；生成时只参考信息架构和 wireframe 规律，不抄具体视觉。

## 使用原则

1. 参考案例只用于学习页面结构、模块顺序、信息密度、CTA 节奏。
2. 输出必须是默认 828px 宽移动端长图 wireframe，不是最终视觉稿。
3. KV 默认作为第一屏头图放进原型图；不要在后续模块重复放置 KV。
4. 原型图可以写模块名称、内容说明、占位框、按钮文案、表格字段、榜单字段。
5. 原型图不要出现“安全区”“40px”“80px”“切口”“续画”“拼接”等生产术语。
6. 段间切口只在蓝图字段中规划，不在 HTML 可视内容里标注。

## 案例 01：通用 Wireframe 模板库

来源：Moqups Wireframe Templates and Examples  
链接：https://moqups.com/templates/wireframes-mockups/

可借鉴点：

- 用灰阶块、标题、段落线、按钮、卡片和图片占位表达页面骨架。
- 适合做活动 H5 的低保真基础模块库。
- 模板覆盖 event landing page、FAQ、pricing、mobile app landing page 等类型。

适合迁移到商单 H5 的模块：

- 顶部活动标题区。
- 主视觉占位区。
- 核心利益点三卡片。
- 嘉宾/角色/奖品卡片横排。
- FAQ 折叠列表。
- 底部 CTA。

生成建议：

- 如果活动信息较少，采用“Hero + 3 个卖点 + CTA + FAQ”。
- 如果活动信息较多，采用“Hero + 活动机制 + 奖励 + 任务 + 榜单 + FAQ + CTA”。

## 案例 02：移动端 UI/UX 截图库

来源：Mobbin mobile screens / flows  
链接：https://mobbin.com/explore/mobile  
链接：https://mobbin.com/explore/mobile/flows/onboarding

可借鉴点：

- 移动端页面通常通过卡片、导航、步骤、表单、列表、浮动 CTA 组织内容。
- Onboarding 类页面强调“第一屏讲清价值，后续逐步降低理解成本”。
- 真实产品页面常用固定底部操作、步骤引导和卡片化信息。

适合迁移到商单 H5 的模块：

- 顶部状态/导航占位。
- 主任务进度条。
- 任务步骤卡。
- 奖励解锁列表。
- 参与按钮或分享按钮。
- 用户状态卡，如“今日进度”“可领取奖励”。

生成建议：

- 对游戏、挑战赛、版本活动类 H5，优先使用“任务进度 + 奖励阶梯 + 榜单”的信息结构。
- 不要让页面只像普通海报，应体现交互页面的模块逻辑。

## 案例 03：落地页案例库

来源：Page Flows landing page examples  
链接：https://pageflows.com/landing-page-design/

可借鉴点：

- 落地页的核心是转化路径：看见主题、理解利益、建立信任、执行操作。
- 页面常有明确 CTA 节奏，首屏、内容中段和底部都可出现 CTA。
- 适合拆解商业推广页的信息顺序。

适合迁移到商单 H5 的模块：

- 活动主张。
- 核心卖点。
- 参与理由。
- 任务/规则。
- 奖励证明。
- CTA。

生成建议：

- 商单 H5 不要只堆模块，要让用户从“为什么参与”走到“怎么参与”。
- CTA 可以在首屏、中段、底部重复，但原型图中用按钮占位表达即可。

## 案例 04：移动 App 落地页结构

来源：Unbounce app landing page examples  
链接：https://unbounce.com/landing-page-examples/app-landing-pages/

可借鉴点：

- 移动 App 落地页常强调下载、注册、预约、加入活动等转化动作。
- 内容需要把“产品/活动价值”和“行动入口”靠前放。
- 图片占位常配合简短说明，不宜大段文字堆叠。

适合迁移到商单 H5 的模块：

- 首屏版本活动主标题。
- 一句话活动利益。
- 任务入口按钮。
- 活动亮点卡。
- 奖励图标占位。
- 规则说明。

生成建议：

- 如果活动目标是召回或促活，页面应把“奖励”和“参与路径”放在前半段。
- 如果活动目标是展示新版本内容，页面应先讲版本亮点，再讲任务奖励。

## 案例 05：中文 H5 活动原型示例

来源：CSDN H5 原型图文章  
链接：https://blog.csdn.net/aimeeth/article/details/131848280

可借鉴点：

- H5 活动原型会把运营模块直接标注在画面里，例如活动主题、规则、榜单、投票、任务入口。
- 对业务方来说，原型图需要可读，能看懂每个模块的目的。
- 可包含活动机制说明、数据字段、按钮、卡片、说明文案。

适合迁移到商单 H5 的模块：

- 活动标题和时间。
- 任务规则。
- 进度/投票/榜单。
- 奖励说明。
- 用户操作入口。
- 活动底部规则。

生成建议：

- 中文商单原型可以更“运营化”，即模块命名清楚，文案不是 lorem ipsum。
- 但视觉仍保持低保真，不要做成最终海报。

## 案例 06：活动落地页 Wireframe 演进

来源：Medium / SaaS event landing page wireframe case study  
链接：https://medium.com/design-bootcamp/case-study-designing-a-landing-page-for-a-virtual-saas-event-64daa2edc50

可借鉴点：

- 活动落地页可以按 AIDA 思路组织：Attention、Interest、Desire、Action。
- 页面从 hero 到亮点、议程、嘉宾、票种/奖励、底部 CTA，形成清晰转化链路。
- 长页面适合用横向分区、卡片阵列和列表来降低阅读压力。

适合迁移到商单 H5 的模块：

- Attention：版本挑战赛主题。
- Interest：版本亮点和玩法说明。
- Desire：奖励阶梯、排行榜、限定权益。
- Action：立即参与、分享、邀请好友。

生成建议：

- 对 PUBG 版本挑战类活动，建议采用“战场气氛首屏 + 任务挑战 + 奖励阶梯 + 榜单 + 分享召回 CTA”。

## 案例 07：Figma Wireframe Kit

来源：Figma Wireframe Kits  
链接：https://www.figma.com/templates/wireframe-kits/

可借鉴点：

- 组件化 wireframe 更适合自动生成：标题、正文、链接、图片、按钮、列表等模块可以稳定复用。
- 知识库可以把模块规范沉淀成“组件字典”。

适合迁移到商单 H5 的模块组件：

- `hero_banner`
- `activity_intro`
- `benefit_cards`
- `task_steps`
- `reward_ladder`
- `ranking_table`
- `ugc_topic_cards`
- `faq_list`
- `footer_cta`

生成建议：

- 蓝图阶段先输出模块组件列表，再由 HTML 阶段按组件生成页面。
- 蓝图先组织完整模块；渲染后再按最大段高自动切成 1-N 段。

## 给 LLM 的案例选择规则

当活动类型是游戏版本、挑战赛、任务活动：

- 优先参考“中文 H5 活动原型示例”“活动落地页 Wireframe 演进”“移动端 UI/UX 截图库”。
- 页面结构：Hero、版本亮点、任务机制、奖励阶梯、榜单、话题分享、FAQ、底部 CTA。

当活动类型是新品发布、品牌推广：

- 优先参考“通用 Wireframe 模板库”“落地页案例库”“移动 App 落地页结构”。
- 页面结构：Hero、卖点、场景、权益、证明、CTA、FAQ。

当活动类型是知识说明或报名页：

- 优先参考“活动落地页 Wireframe 演进”“Figma Wireframe Kit”。
- 页面结构：Hero、价值说明、流程、议程/规则、报名表单或 CTA、FAQ。

## PUBG 版本挑战赛推荐原型结构

页面尺寸：

- 宽度：默认 828px。
- 目标高度：4200px 到 5000px。
- 切片：由完整页面高度自动规划为 1-N 段。

模块组 A（开场，不代表固定切片边界）：

- 顶部状态栏和活动标题。
- 主视觉占位：PUBG MOBILE 版本挑战赛。
- 活动时间和一句话利益点。
- 版本亮点模块：新战场、新任务、组队挑战。
- 参与入口按钮占位。

模块组 B（核心机制，不代表固定切片边界）：

- 任务流程：登录、完成挑战、邀请好友、领取奖励。
- 奖励阶梯：基础奖励、进阶奖励、排行奖励。
- 今日进度卡：任务完成度、可领取奖励。
- 榜单预览：排名、昵称、积分/任务数。

模块组 C（收口，不代表固定切片边界）：

- UGC/话题分享：晒战绩、组队招募、版本探索。
- FAQ：如何参与、奖励何时发放、投稿要求。
- 活动规则和免责声明。
- 底部固定或大按钮 CTA：立即加入版本挑战。

候选裁切建议：

- 将“版本亮点”与“任务流程”之间的自然留白作为候选 safe band。
- 将“榜单预览”与“UGC/话题分享”之间的自然留白作为候选 safe band。
- 候选区域数量可以多于最终切点；最终 cuts 由 ImageService 根据页面高度和视觉复杂度选择。
- 不要在画面里显示切片、安全区、40px、80px 等字样。
