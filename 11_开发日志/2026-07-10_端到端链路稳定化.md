# 端到端链路稳定化

日期：2026-07-10

## 本轮目标

本轮进入第十三步，重点不是接真实生图，也不是重做 UI，而是把当前工作台主链路跑顺：

```text
Home 输入需求
-> 创建当前项目
-> 进入 Project Workspace
-> 原型预览
-> Inspector 人工编辑
-> 保存回正式 h5-artifact.json
-> 确认原型
-> 生成长图 mock
-> 导出 mock
```

## 本次完成

- 前端 Inspector 保存前会先拉取 `/api/projects/current` 最新状态，再基于 daemon 最新正式 Artifact 生成 patch。
- 如果浏览器旧状态里的 `selectedSectionId` 已不存在，会自动同步 daemon 最新项目，并中止保存，避免继续 patch 已删除或已替换的 section。
- 新增 daemon API：`POST /api/projects/current/artifact/confirm`。
- 确认原型时会校验当前正式 `h5-artifact.json`，写入 `confirmedAt`、`confirmedVersionId`，并备份上一版为 `h5-artifact.previous.json`。
- 确认成功后会更新 `run-state.json`，将 `prototype_review` 置为 `done`。
- 确认成功后会更新 `files.manifest.json`，并重新生成 `prototype.html`。
- `long_image_generation` mock run 已增加后端门槛：只有正式 Artifact 状态为 `confirmed` 时才能启动。
- 未确认原型时启动长图会返回 409，并写入 blocked/error 事件，前端会提示「请先确认原型」，不会再降级成本地 mock。
- 确认后启动长图 mock run 可以完整跑完，`slice-plan.json` 会写入 `confirmedVersionId`。
- `output/long-image.png` 和 `output/export.zip` 仍是 JSON placeholder，字段中明确标注 `placeholder: true` 和 `mock: true`，不伪装成真实图片或 ZIP。
- `prototype.html` 落盘模板已跟随 hidden section 逻辑，隐藏模块不会继续出现在落盘 HTML 中。

## 当前 prototype.html 状态

当前 `prototype.html` 仍是 daemon 根据正式 `h5-artifact.json` 生成的简易 HTML 模板。

它能用于：

- 验证正式 Artifact 是否已落盘。
- 验证人工编辑是否会刷新原型预览来源。
- 作为后续接入更高质量原型生成器的文件占位和协议入口。

它还不能代表最终高质量 H5 原型。旧项目里的 HTML wireframe 生成、组件规则、模块边界标记和渲染切片能力仍未接入当前工作台。

## 下一步建议

1. 先把当前 `H5Artifact` 和旧 Python pipeline 的输入输出做一层 adapter 设计，不要直接调用旧 runner。
2. 把旧项目的高质量 HTML wireframe 生成能力拆成可独立 runner，再接入 daemon Runner Adapter。
3. 在前端增加文件预览或 run detail，便于查看 `prototype.html`、`slice-plan.json` 和 placeholder 输出。
4. 后续真实导出时，让 `export.zip` 包含 `brief.md`、`h5-artifact.json`、`prototype.html`、`slice-plan.json`、长图和质检报告。
