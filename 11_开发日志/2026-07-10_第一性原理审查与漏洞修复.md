# 第一性原理审查与漏洞修复

日期：2026-07-10

## 本轮目的

本轮没有增加新的产品方向，而是从「用户到底在确认什么、系统到底用什么生成、磁盘到底保存了什么」三个基本问题出发，对现有工作台做对抗式审查并直接修复。

产品结构保持不变：

```text
一级 Home
-> 二级 Project Workspace
-> 候选原型 Review & Apply
-> 人工编辑
-> 确认原型
-> 长图 mock run
-> mock 导出
```

本轮没有接真实生图服务，没有运行 `run_local_h5.py`，没有引入账号、团队协作、支付或自由画布。

## 已解决的问题

### 1. 候选 HTML 与正式 Artifact 分裂

旧逻辑只把 `prototype.candidate.html` 覆盖为正式 `prototype.html`，正式 `h5-artifact.json` 仍保留 4 模块；确认原型时还可能用旧 Artifact 重新生成 HTML。

现在：

- Apply 会同时读取候选 HTML 和 `legacy-blueprint.candidate.json`。
- blueprint 会映射成正式 H5Artifact。
- 正式 HTML、blueprint、Artifact 记录同一来源和 SHA-256。
- 当前项目已从旧 4 模块 Artifact 对账为 9 模块 Artifact。
- 旧 Artifact 已备份为 `h5-artifact.pre-reconcile.json`。
- 确认 legacy 原型不会再覆盖正式 HTML。

### 2. 用户查看后候选文件可能被替换

前端查看候选时会记录内容哈希，Apply 请求必须带 `expectedHash`。daemon 会重新计算文件哈希；查看后内容若有变化，Apply 会拒绝并要求重新查看。

### 3. legacy 文件可被无提示篡改

确认原型前会校验：

- 正式 `prototype.html` 哈希是否等于已应用候选哈希。
- legacy blueprint 哈希是否等于应用时哈希。
- 必需素材状态是否有效。
- 必需素材文件是否真实存在于当前项目目录。

隔离测试中手工修改正式 HTML 后，确认接口正确返回 400。

### 4. dry-run 误入正式生产链

`prototype.dry-run.html` 现在只用于技术验证：

- 可以查看。
- 可以标记放弃。
- 不允许 Apply 为正式原型。

正式主线只接受 Home 生成的 product candidate。

### 5. KV 只有文件名，没有真实文件

Home 的 KV 选择器现在读取真实文件内容并发送给 daemon。daemon 会：

- 校验 MIME。
- 限制单文件 12MB。
- 写入 `current/assets/kv_main.*`。
- 更新正式 H5Artifact 的 ImageSlot。
- 把真实 KV 复制给 legacy runner 输入。

没有真实 KV 时，Home 候选生成返回 422，不再使用占位图冒充用户素材。

### 6. 素材/KV 页面只改浏览器状态

6 类 ImageSlot 已接入真实 daemon 上传、替换和清除 API。素材变化会使已确认/已生成 Artifact 回到 `edited`，要求重新确认。

### 7. 组件替换和主题修改无法保存

Artifact patch 协议新增：

- `replace_section_component`
- `update_theme_style`
- `update_image_slot`

Inspector 的组件替换和主题风格会写回正式 `h5-artifact.json`，不再只存在于 React state。

### 8. 多项目 tab 是浏览器假状态

项目存储新增：

- `current/`：当前打开项目。
- `snapshots/<projectId>/...`：历史项目快照。
- `GET /api/projects`：列出磁盘项目。
- `POST /api/projects/:id/open`：打开磁盘项目。

新建或切换项目会先安全轮转 current，项目 tab 不再依赖浏览器伪造完整项目状态。

### 9. 两套 daemon API 可绕过确认门槛

旧的按 projectId 直接更新 Artifact、确认、生成 SVG 和导出接口已退役并返回 410。当前只保留：

- current project API
- Artifact patch / confirm API
- Runner API
- SSE 事件流

长图流程不能再从旧接口绕开确认门槛。

### 10. run 并发和重启恢复不可靠

- 已有 run 执行时，新 run 返回 409。
- run 执行时不能切换或替换当前项目。
- 每次重新运行长图时会重置下游阶段，不继承上一轮 done 状态。
- daemon 启动时会把上次中断的 `running` 阶段和文件恢复为 `error`。
- 生成中的 Artifact 会恢复为已确认或已编辑状态。

### 11. 磁盘损坏会静默覆盖项目

`loadProjectState()` 现在执行完整 H5Artifact 校验。读取失败时：

- 不覆盖现有项目。
- 写入 `recovery-report.json`。
- daemon 明确失败，要求人工恢复或使用 snapshot。

损坏的 `events.jsonl` 行会被跳过，并写入 `events.recovery.json`。

### 12. 本地 daemon 和 runner 安全边界过宽

- daemon 继续只监听 `127.0.0.1`。
- CORS 从 `*` 改为工作台本地 Origin 白名单。
- JSON 请求体限制为 16MB。
- Codex / Python runner 不再继承完整 `process.env`，改用环境变量白名单。
- 子进程中止或超时时先 SIGTERM，超时后 SIGKILL。

第一阶段仍不增加账号鉴权；对本地恶意进程的防护不属于当前个人工具边界。

### 13. 旧文件可能被误认成新产物

- 不在 FileManifest 的历史文件不能通过文件 API 读取。
- `waiting` 状态的 slice plan、长图和导出文件返回 404。
- mock 输出内容明确为 JSON metadata，包含 `placeholder: true` 和 `mock: true`。
- 前端导出只下载 `export-placeholder.json`，不再把浏览器状态改成 done 或伪装真实 ZIP。

### 14. 没有自动回归测试

新增 `pnpm test`，当前 3 个测试覆盖：

- H5Artifact 深层校验和必需素材确认门槛。
- legacy 候选 HTML + blueprint 同源 Apply，确认前后 HTML 哈希保持一致。
- 多项目 snapshot 列表与重新打开。

## 尚未完成

- 没有运行 `run_local_h5.py`。
- `long-image.png` 和 `export.zip` 仍是明确标注的 mock metadata。
