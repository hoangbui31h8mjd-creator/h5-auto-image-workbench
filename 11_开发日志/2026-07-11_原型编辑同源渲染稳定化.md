# 2026-07-11 原型编辑同源渲染稳定化

## 本次修改背景

旧 runner 候选原型应用后，正式 `prototype.html` 的来源会是 `legacy_runner`。此前 Inspector 保存任何编辑时，`projectStore` 都会把 `prototypeSource` 切回 `artifact_renderer`，再用简易 `renderPrototypeHtml()` 覆盖正式 HTML。这会丢失旧 runner 的 CSS、布局、素材和模块结构，也会破坏候选审稿后的同源关系。

本轮只稳定编辑与渲染边界，不接真实生图，不运行 `run_local_h5.py`，不改变「一级 Home + 二级 Project Workspace」结构。

## 本次目标

- 依据 `prototypeSource.kind` 选择明确的原型渲染策略。
- legacy 原型只做可安全定位的局部 DOM 编辑，不降级为简易模板。
- 保存后更新 HTML 哈希，并保持确认原型的哈希校验可用。
- 对无法可靠局部改写的操作返回 `requires_regeneration`，不改变正式文件。

## 实现结果

### 渲染策略

- `artifact_renderer`：继续使用现有简易 renderer，适用于结构化 Artifact 原型。
- `legacy_runner`：读取正式 `prototype.html`，通过 HTML parser 查找旧 runner 已有的 `data-section` / `data-section-id`、`data-component` 标记。

legacy 策略支持：

- 模块基础文案：优先使用 `data-h5-field` / `data-field` / `data-text-field`，无显式字段时仅安全映射到单一标题节点或正文节点。
- 模块高度：写入受控 `data-workbench-height` 属性，并在新增的 scoped style 中覆盖最小高度，不改旧 CSS。
- 隐藏 / 恢复：写入或移除受控 `data-workbench-hidden` 属性。
- 上移 / 下移：只在所有标记模块处于同一父容器、且可完整分组时重新排序；会连同紧邻的 module-gap 一起移动。

每次 legacy 保存都会先备份：

- `prototype.previous.html`
- `h5-artifact.previous.json`

保存成功后会：

- 保留 `prototypeSource.kind = legacy_runner`。
- 更新 `prototypeSource.sourceHash` 为正式 `prototype.html` 的新 SHA-256。
- 保留 blueprint 文件和 blueprint 哈希。
- 在 FileManifest 保持 `prototype.html` 为 `legacy_applied`，不触发后续简易 renderer 覆盖。
- 写入 `file_written: prototype.previous.html`、`file_written: prototype.html`、`artifact_updated`。

### 明确不支持的编辑

下列操作不会静默降级，也不会写正式 HTML / Artifact：

- 全局主题：`update_theme_style`
- 全局 Artifact 标题：`update_artifact_title`
- 组件替换：`replace_section_component`
- 任意 Props：`update_section_prop`
- 图片槽位映射：`update_image_slot`
- 无法在旧 HTML 中唯一定位的模块或文本字段
- 不处于同一安全父容器的排序

这些情况返回 HTTP 409 和 `error.code = requires_regeneration`。前端会提示需要重新生成候选原型；正式文件保持不变。
