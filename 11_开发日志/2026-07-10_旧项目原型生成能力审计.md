## 总结结论

旧项目里确实沉淀了有价值的原型生成和分段生图能力，但它不是一个可直接接入当前工作台的标准 Codex Skill。

当前可复用资产分成三类：

- 可复用知识：执行版 Skill 文档、组件 registry、H5 原型生成知识库、Figma 解析规格。
- 可复用流程：需求输入、蓝图 JSON、HTML wireframe、渲染、智能切片、质检、mock/真实 Image2、拼接和输出目录结构。
- 可复用代码：旧代码项目里的 `h5_agent/` Python 分层模块和 runner。

当前不能直接复用的部分：

- 旧项目没有标准 `SKILL.md` 可直接作为 Codex skill 安装运行。
- 桌面旧资料项目主要是产品包、规则、归档和输出，不是当前工作台 daemon 可直接调用的 npm workspace 包。
- Dify / ComfyUI 归档链路是历史验证资产，不应重新成为新工作台第一阶段主入口。
- 旧 Python runner 的输入输出仍是任务目录和文件产物，需要先做 adapter，不能直接替换当前 `H5Artifact` 协议。

## 原型生成流程

旧代码项目中存在可运行的本地原型生成流程：

- `h5_agent/runner/run_local_prototype.py`
- `h5_agent/prototype/local_generator.py`
- `h5_agent/render/local_renderer.py`
- `h5_agent/slicing/local_slicer.py`
- `h5_agent/quality/local_report.py`

`run_local_prototype.py` 的链路是：

```text
任务目录
-> load_task_input
-> build_local_prototype
-> 01_blueprint.json
-> 02_prototype.html
-> render_prototype_html
-> 03_prototype_full.png
-> slice_rendered_prototype
-> 04_slices/
-> 07_quality_report.md
-> run_summary.json
```

`local_generator.py` 是规则型 MVP 生成器，不调用外部 LLM。它会读取 skill bundle 和组件 registry，生成 deterministic blueprint 和 HTML wireframe。

这套能力比当前新工作台的 `prototype.html` 简易模板更接近“可用于切片和生图的原型”，但仍需要 adapter 才能接入当前工作台。

## runner / pipeline

旧代码项目中存在完整本地 runner：

- `h5_agent/runner/run_local_prototype.py`
- `h5_agent/runner/run_local_h5.py`

`run_local_h5.py` 在原型链路后继续执行：

```text
prototype generation
-> render
-> slicing
-> generate_h5_visual
-> generated segments
-> final long image
-> generation debug
-> full run summary
```

它依赖：

- `h5_agent/generation/local_image2.py`
- `h5_agent/storage/local_store.py`
- `h5_agent/slicing/local_slicer.py`
- `app.services.renderer`
- `app.services.image_ops`

当前可以作为未来 daemon Runner Adapter 的候选后端，但不应在本轮直接接入。

5. 所有旧 runner 输出必须写入新项目 `.h5-workbench/projects/current/` 或后续项目目录，不写回旧项目目录。
