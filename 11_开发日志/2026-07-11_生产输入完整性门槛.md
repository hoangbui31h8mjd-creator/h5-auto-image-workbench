# 第十九步：生产输入完整性门槛

日期：2026-07-11

## 本轮目标

本轮只加固候选审核、素材可信来源、H5Artifact 语义完整性，以及「确认原型 -> 长图 mock」之间的生产输入门槛。

产品结构没有改变，仍为：

```text
一级 Home
-> 二级 Project Workspace
-> 人工审核 / 编辑 / 确认
-> 长图 mock
```

本轮没有接真实生图，没有运行 `run_local_h5.py`，没有生成真实 PNG 长图或真实 ZIP 导出包。

## 已完成

### 1. 候选 HTML + blueprint 双哈希

- 新增 `GET /api/prototypes/candidate/review`。
- 查看候选时同时返回：
  - `candidateHtmlHash`
  - `blueprintHash`
- `POST /api/prototypes/candidate/apply` 现在必须同时提交：
  - `expectedCandidateHtmlHash`
  - `expectedBlueprintHash`
- 缺少任一有效 SHA-256 返回结构化 `candidate_hash_required`。
- 查看后 HTML 或 blueprint 任一变化，返回 `409 candidate_changed`。
- Apply 不再校验后重新复制磁盘文件；正式 `prototype.html`、candidate 文件和 applied blueprint 均写入刚完成双哈希校验的同一份内存 Buffer。
- 新增稳定正式 blueprint：
  - `legacy-task/applied/legacy-blueprint.applied.json`
- legacy `prototypeSource.blueprintFile` 不再指向可能被下一次候选生成覆盖的 candidate blueprint。
- Apply 使用多文件原子提交；哈希失败或提交失败时，正式 HTML、Artifact、manifest 和两个 previous 备份保持不变。
- 前端查看候选后保存两个哈希；Apply 同时提交两个哈希。收到 `candidate_changed` 后清空旧候选和哈希，要求重新查看。

### 2. 素材可信元数据

H5Artifact 的 managed asset 现包含：

- `assetId`
- `slotId`
- `relativePath`
- `sha256`
- `mimeType`
- `byteSize`
- `originalFilename`
- `uploadedAt`
- `verificationStatus`

素材写入与确认门槛：

- data URL MIME、扩展名和前端 File MIME 不再作为事实源。
- daemon 从真实字节识别 PNG / JPEG / WebP / SVG / 文本类型。
- 声明 MIME 与真实字节不一致会拒绝。
- 只允许 `assets/` 下的安全相对路径。
- 拒绝 `brief.md`、JSON 冒充图片、目录、`../` 越界路径和 symlink 越界。
- 确认时重新读取文件，复核普通文件、真实路径、MIME、字节数和 SHA-256。
- `ImageSlot` 的 uploaded / locked / generated 状态必须引用受管理的 `assetId`；不能只写任意 `assetPath`。
- 通用 Artifact patch 的 `update_image_slot` 只保留 empty / missing，不能伪造 uploaded / locked / generated。
- 旧素材若只有 `assetPath`，确认前会先安全读取和验证；验证成功后迁移 managed metadata，迁移写入前的旧 Artifact 保存在 `h5-artifact.previous.json`。验证失败则不迁移、不确认。

### 3. H5Artifact 语义校验

新增并接入 `validateH5ArtifactIntegrity()`，实际校验：

- section `id` 唯一。
- section `sectionId` 唯一。
- `order` 唯一、按数组顺序从 1 连续。
- `component` 必须存在于当前组件 registry。
- section `imageSlotIds` 必须引用顶层真实槽位。
- `usedBySections` 必须引用真实 section。
- uploaded / locked / generated 槽位必须引用真实 managed asset。
- managed asset 与槽位的 slotId、相对路径和 assetId 必须一致。
- `pageSize.height` 必须等于全部可见 section 高度之和。
- legacy `prototypeSource` 必须有合法 sourceHash、blueprintFile 和 blueprintHash。

语义失败统一抛出结构化：

```text
code = artifact_integrity_failed
```

Artifact draft apply、候选 apply、人工 patch、素材保存和 confirm 均已接入语义校验，失败时不写正式文件。

### 4. production-integrity 与长图前置门槛

确认原型现在写入：

```text
production-integrity.json
```

清单记录：

- projectId / artifactId / confirmedVersionId
- 正式 `h5-artifact.json` SHA-256
- 正式 `prototype.html` SHA-256
- legacy applied blueprint SHA-256（legacy 来源时）
- 必需素材的 assetId、slotId、路径、SHA-256、MIME 和字节数

`long_image_generation` 启动前会重新读取并复核该清单。以下任一变化都会返回：

```text
HTTP 409
code = production_integrity_failed
```

- 正式 HTML 变化
- 正式 Artifact 变化
- legacy blueprint 变化
- 必需素材变化
- daemon 内存 Artifact 与磁盘 Artifact 不一致

`mockLongImageRunner` 现在必须收到已复核的 `ProductionIntegrityRecord` 才能启动。runner 不再把正式 Artifact 状态改为 generating / generated，避免运行过程自己破坏已确认输入；进度只写 RunState 和 FileManifest。

## 第二轮对抗式自审

### 自审发现

1. 第一轮实现后发现旧 `mockLongImageRunner` 仍会把正式 Artifact 改成 generating / generated，这会让确认清单在运行开始后立刻失效。
2. 发现 legacy `prototypeSource.blueprintFile` 仍指向 candidate 目录，下一次候选生成可能覆盖正式来源。
3. 初版测试虽覆盖双哈希变化，但缺候选磁盘提交失败回滚、旧素材迁移前备份、前端双哈希请求体三个直接证据。
4. 发现历史 fixture 的空 `copy_reference` 槽位仍带过期 `rules_1` 引用。该兼容数据只在确认/写入前做窄范围清理；任意其他无效 usedBySections 仍由语义校验拒绝。

### 自审修复

- 删除 mock 长图 runner 对正式 Artifact status 的修改。
- 候选 Apply 时固化 `legacy-blueprint.applied.json`，正式 prototypeSource 改指向 applied 文件。
- 增加候选提交故障回滚测试。
- 增加旧素材安全迁移与 `h5-artifact.previous.json` 迁移前内容测试。
- 增加前端 API client 双哈希请求体测试。
- 增加确认提交失败回滚、symlink 越界、daemon 重启后完整性复核和真实 9 模块 snapshot 只读复制测试。
